# GardenOS V2 Features — Design Spec

> 6 independent features to add after the MVP has been running. Each can be built and shipped independently.

**Nav change:** Bottom tab bar evolves incrementally:
- **With Seed Inventory (feature 4):** Add 5th "Seeds" tab. Keep "Beds" and "Succession" labels unchanged.
- **With Succession Gantt (feature 5):** Rename "Succession" tab to "Plan".
- **With Bed Designer (feature 6):** Rename "Beds" tab to "Garden", route changes from `/beds` to `/garden` (old route redirects).

---

## 1. Harvest Log

**Purpose:** Track what's harvested, when, and roughly how much. Feeds AI end-of-season variety comparison.

### Data Model

New `harvests` table:

```
harvests
├── id (primary key)
├── plant_id (FK → plants, not null, ON DELETE CASCADE)
├── date (date, not null, default today)
├── quantity (string, not null) — enum: small, medium, large, huge
├── notes (text, optional)
├── created_at
```

FK uses `ON DELETE CASCADE` — if a plant is deleted between seasons, its harvest history goes with it. This matches the pattern used by `stage_histories`.

### Routes

| Method | Path | What |
|--------|------|------|
| POST | `/plants/:id/harvests` | Log a harvest |
| GET | `/api/plants/:id/harvests` | JSON harvest history for a plant |

### UI

- **Plant detail page:** "Log harvest" button below the actions card
  - Tap → inline form expands: 4 quantity buttons (small/medium/large/huge) + optional note field + Save
  - After save: new harvest entry appears in the timeline
- **Plant detail timeline:** Harvest entries interleaved with stage transitions, marked with a harvest icon
- **AI context:** `AIAdvisoryService.build_context` includes harvest counts per plant for end-of-season summaries

### Empty State

"No harvests logged yet."

## 2. Photo Journal

**Purpose:** Snap photos, tag to plants/stages. Build a visual timeline per plant.

### Data Model

New `photos` table:

```
photos
├── id (primary key)
├── plant_id (FK → plants, optional, ON DELETE SET NULL)
├── bed_id (FK → beds, optional, ON DELETE SET NULL)
├── lifecycle_stage (string, optional) — auto-set from plant's current stage
├── filename (string, not null) — relative path under public/uploads/
├── caption (text, optional)
├── taken_at (datetime, default now)
├── created_at
```

### Storage

- Directory: `public/uploads/YYYY/MM/` — organized by month
- Thumbnails: generate 200px-wide thumbnail on upload via ImageMagick (`convert -resize 200x`) or skip thumbnails if ImageMagick unavailable (serve originals, let browser resize)
- Max upload size: 10 MB per photo. Server rejects larger files.
- On `DELETE /photos/:id`: delete both the DB record AND the file from disk.
- No background cleanup — single user, disk is cheap.

### Routes

| Method | Path | What |
|--------|------|------|
| POST | `/plants/:id/photos` | Upload photo for a plant |
| GET | `/plants/:id/photos` | Photo gallery for a plant |
| DELETE | `/photos/:id` | Delete a photo |

### UI

- **Plant detail page:** Camera icon button next to "Log harvest"
  - Tap → file picker (phone opens camera on mobile)
  - Auto-tags `plant_id` and current `lifecycle_stage`
  - Optional caption field
- **Plant timeline:** Photos appear as thumbnail cards alongside stage transitions and harvests
- **No standalone gallery page** — photos always in context of a plant
- **No bed photo upload for now** — `bed_id` column exists for future use but no route to populate it

### Empty State

"No photos yet. Tap the camera icon to add one."

## 3. Sensor Integration

**Purpose:** Pull Hydrawise irrigation data + indoor temp sensors from HA. Auto-skip watering tasks. Show zone status on dashboard.

### Data Model

No new tables. Sensor data is polled live, not stored.

### Configuration

New env vars in `.env`:

```bash
# Comma-separated HA entity IDs for Hydrawise zones
HA_HYDRAWISE_ZONES=switch.hydrawise_zone_1,switch.hydrawise_zone_2

# Indoor temperature sensor (heat mat area)
HA_INDOOR_TEMP_ENTITY=sensor.indoor_temperature

# Rain sensor or weather-based skip
HA_RAIN_SENSOR=binary_sensor.hydrawise_rain_sensor
```

### Architecture

New `SensorService` class:

```ruby
class SensorService
  def self.fetch_zones     # → [{name, state, next_run}]
  def self.fetch_indoor_temp  # → {temp, entity_id} or nil
  def self.rain_detected?  # → boolean
end
```

Uses same curl-based HA API pattern as WeatherService.

### Task Auto-Skip

Enhance `TaskGenerator`:
- Before generating watering tasks, check `SensorService.rain_detected?` and Hydrawise zone status
- If irrigation is active or rain detected, mark existing watering tasks as "skipped" with note "Auto-skipped: irrigation active" or "Auto-skipped: rain detected"

### Dashboard UI

New sensor strip on the dashboard (between alert banner and summary strip):
- Hydrawise zones: green dot = idle, blue dot = running, gray = offline
- Indoor temp: current reading + warning if outside 25-32°C range (heat mat sweet spot)
- Only renders if `HA_HYDRAWISE_ZONES` or `HA_INDOOR_TEMP_ENTITY` are configured

### Empty State

Sensor strip hidden entirely when no sensor env vars are set.

## 4. Seed Inventory

**Purpose:** Track seed packets — what you have, how much is left, when it expires.

### Data Model

New `seed_packets` table:

```
seed_packets
├── id (primary key)
├── variety_name (string, not null)
├── crop_type (string, not null)
├── source (string) — "Loukykvět", "Magic Garden Seeds", "saved"
├── quantity_remaining (integer) — packets or approximate seed count
├── sow_by_date (date, optional)
├── purchase_date (date, optional)
├── url (string, optional) — link to supplier product page
├── notes (text, optional)
├── created_at
├── updated_at
```

### Routes

| Method | Path | What |
|--------|------|------|
| GET | `/seeds` | Seed inventory list |
| GET | `/seeds/:id` | Seed packet detail |
| POST | `/seeds` | Create seed packet |
| PATCH | `/seeds/:id` | Update seed packet |
| DELETE | `/seeds/:id` | Delete seed packet |
| GET | `/api/seeds` | JSON seed inventory |

### UI

- **New "Seeds" tab** in bottom nav (5th tab, between Garden and Plan)
- **List view:** Grouped by crop type, each card shows variety + source + quantity + sow-by status
- **Sow-by warnings:** Amber pill if within 6 months of expiry, red pill if expired
- **Add form:** Simple form at top or via "+" button
- **Link to plants:** When creating a Plant record, optional dropdown to link to a seed packet. On sow, auto-decrement `quantity_remaining` by 1. If quantity reaches 0, show "Out of stock" badge but don't block sowing (you might have more than you tracked). Negative values are allowed.

### Empty State

"No seed packets tracked yet. Add your first packet."

## 5. Succession Gantt

**Purpose:** Visual interactive timeline showing all succession sowings — past, current, future.

### Data Model

No new tables — reads existing `SuccessionPlan` and `Task` records.

### Routes

| Method | Path | What |
|--------|------|------|
| GET | `/succession` | Gantt view (replaces current page) |
| PATCH | `/tasks/:id/reschedule` | Update a task's due_date (for drag). Only changes the individual task's date — does NOT modify the SuccessionPlan interval. This allows manual adjustments while keeping the plan template intact for future task generation. |

### UI — Alpine.js + CSS Grid

**Layout:**
- Left column: crop names (one row per SuccessionPlan)
- Right area: scrollable timeline with CSS Grid columns (one per day or week)
- Header: week/month markers
- Today marker: vertical red/green line

**Bars:**
- Each sowing = a bar spanning its due_date
- Bar colors: green (task status "done"), amber (status "upcoming" + due within 7 days), gray (future)
- Bar label: "Sow #N" inside the bar

**Interactions (Alpine.js):**
- **Drag bar** to move horizontally → updates due_date via PATCH `/api/tasks/:id/reschedule`
- **Click bar** → shows popover with sowing details (variety, beds, notes) + "Mark done" button
- **Scroll** horizontally through the timeline
- **Zoom:** toggle between day/week view

**Implementation approach:**
- Alpine.js `x-data` manages the Gantt state: crops, bars, scroll position, zoom level
- Each bar is a `<div>` positioned absolutely within the grid
- Drag uses `@mousedown`/`@touchstart` + `@mousemove`/`@touchmove` on the bar, updating `left` position
- On `@mouseup`/`@touchend`, calculate new date from pixel position, PATCH to server

### Empty State

"No succession plans set up. Create one to see the timeline."

## 6. Bed Designer

**Purpose:** Draw the garden layout visually — place beds, name them, assign rows and slots. Then overlay what's planted where.

### Data Model

Modify existing `beds` table — add spatial columns:

```
beds (modified)
├── ... existing columns ...
├── canvas_x (float) — X position on canvas
├── canvas_y (float) — Y position on canvas
├── canvas_width (float) — width on canvas
├── canvas_height (float) — height (for rectangles)
├── canvas_points (text) — JSON array of [x,y] for polygons, null for rectangles. Model gets `canvas_points_array` helper (JSON parse/serialize, same pattern as `SuccessionPlan#varieties_list`).
├── canvas_color (string) — fill color override, optional
```

New migration adds these columns with defaults (null = not yet placed on canvas). A bed with all canvas fields null simply hasn't been placed on the visual canvas yet — it still works in the data grid.

### Routes

| Method | Path | What |
|--------|------|------|
| GET | `/garden` | Bed designer / visual map |
| PATCH | `/api/beds/:id/position` | Update bed canvas position/size |
| POST | `/api/beds` | Create a new bed |
| PATCH | `/api/beds/:id` | Update bed properties |

### UI — Alpine.js + Inline SVG

**Canvas:**
- Full-width SVG element with a grid background (snap lines)
- Beds rendered as SVG `<rect>` (rectangles) or `<polygon>` (irregular shapes)
- Arches rendered as SVG `<line>` or `<path>` between beds
- Grid size: configurable (default 0.5m per cell)

**Drawing tools (toolbar at top):**
- **Select** (default) — click to select a bed, drag to move, corner handles to resize
- **Rectangle** — click and drag to draw a new rectangular bed
- **Polygon** — click points to place vertices, double-click or click start point to close the shape
- **Line** — click two points to draw an arch/path between beds

**Selected bed panel (slides up from bottom or sidebar on tablet):**
- Bed name (editable)
- Dimensions (auto-calculated from canvas)
- Rows + slots editor (add/remove/rename)
- Bed type, orientation, wall type

**Plant overlay (toggle button):**
- When enabled: beds fill with color-coded crop indicators
- Same color scheme as current bed map (red=tomato, orange=pepper, etc.)
- Shows variety abbreviation in each slot position

**Implementation approach:**
- Alpine.js `x-data` manages: tool mode, selected bed, beds array, zoom, pan
- SVG elements with Alpine bindings: `:x`, `:y`, `:width`, `:height`, `:points`
- Drag: `@mousedown` on bed → track mouse delta → update x/y → on release PATCH to server
- Resize: corner handles, same pattern
- Polygon: accumulate clicks in `points` array, render `<polygon>` preview, close on double-click
- Grid snap: round all coordinates to nearest grid unit

### Empty State

Empty canvas with grid and a prompt: "Tap + to add your first bed" with the rectangle tool selected by default.

## Navigation Update

Bottom tab bar changes from 4 → 5 tabs:

| Tab | Icon (Lucide) | Route | Label |
|-----|--------------|-------|-------|
| Home | `home` | `/` | Home |
| Plants | `leaf` | `/plants` | Plants |
| Garden | `map` | `/garden` | Garden |
| Seeds | `package` | `/seeds` | Seeds |
| Plan | `gantt-chart` or `calendar-range` | `/succession` | Plan |

The old `/beds` route redirects to `/garden`.

---

## Build Order Recommendation

1. **Harvest log** — smallest, immediately useful, feeds AI
2. **Seed inventory** — small, new page + nav update
3. **Sensor integration** — no new tables, enriches dashboard
4. **Photo journal** — medium, file upload handling
5. **Succession Gantt** — medium, complex Alpine.js interactions
6. **Bed designer** — largest, complex SVG + drawing tools

Each feature is independently shippable.

---

## What's NOT Changing

- Existing Plant, Task, SuccessionPlan, Advisory, StageHistory models — preserved, with additive changes only (new columns on `beds` for canvas, new associations on Plant for harvests/photos)
- Dashboard structure — summary strip + tabs stays, sensor strip added
- AI advisory service — context enriched with harvest data, not restructured
- WeatherService, NotificationService — unchanged
- HACS integration — unchanged (can be extended later to expose new entities)
